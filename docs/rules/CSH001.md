# `CSH001` - Heading hierarchy

Tags: `changeset`

Aliases: `changeset-heading-hierarchy`

This rule is triggered when a changeset file contains an h1 heading or skips heading levels (e.g., jumping from h2 to h4 without an h3 in between).

Changeset files use h2 headings (`##`) as the top-level section delimiter. h1 headings are reserved for the version title that the changelog formatter generates, so they must never appear in changeset source files. Heading levels must also increase sequentially -- you cannot skip from h2 to h4 without an intervening h3.

## Invalid

h1 headings are not allowed:

```markdown
# This is not allowed

## Features

- Added new API endpoint
```

Skipping heading levels is not allowed:

```markdown
## Features

#### This skips h3

- Added new API endpoint
```

## Valid

Headings start at h2 and increase sequentially:

```markdown
## Features

### Subheading

- Added new API endpoint
```

Multiple h2 sections are fine as long as no levels are skipped:

```markdown
## Features

- Added new API endpoint

## Bug Fixes

- Fixed login issue
```

## Fix

- Replace any `#` (h1) headings with `##` (h2) or remove them entirely.
- If heading levels are skipped, add the missing intermediate levels or reduce the depth of the offending heading.

## Rationale

Consistent heading hierarchy makes changeset files predictable for both human reviewers and AI agents that parse them. The changelog formatter relies on h2 headings to identify category sections -- h1 headings and skipped levels break that expectation and produce malformed output.
